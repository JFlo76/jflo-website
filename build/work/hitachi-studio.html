Bringing IT to OT

Hitachi’s Lumada Studio is the user interface for the data collected and analyzed on the Lumada Core platform, and is primarily used by people in the manufacturing industry to monitor production for predictive maintenance.

The goals for this application included an intuitive interface, data visualization through customizable dashboards, a library of widgets for the charts/graphs, and the ability to generate reports.

User Research

Before starting any designs, I needed to fully understand the user profiles for a manufacturing plant. I sought out a few Hitachi OT specialists to gain insight on who would be using this application and what they typically do in their day. After a couple of meetings and sleuthing through Confluence documentation, I discovered there were essentially 3 types of user roles:

Admin Adam:

A technically-inclined power-user who most likely is a lead architect at Hitachi Consulting. He works closely with customers to deploy and configure Studio according to their requirements. He needs to have full access to all aspects of Studio to fulfill his job. Adam has the technical acumen to understand the intricacies of the technologies behind Studio and how to customize it. He also has a strong understanding of all the different Lumada suite of products.

OT Manager Mary:

Mary is an operations manager at Large Rail Operator GmbH. Mary manages a large number of people and multiple production line. She needs to have full visibility of all the operators and needs to be able to manage what her employees have access to on Studio. Mary is not knowledgeable on the technical aspects of Studio but is trained in how to use Studio for her purposes.

  • Checks work orders per asset
  • Checks news/alert feed
  • Concerned with efficiency & degradation
  • Is proactive to prevent problems
  • Also performs personnel admin management

OT Operator Eric:

Eric is an operator for Huge Automotive Ltd. He works in the car assembly process, where he is responsible for assembling the seats on the car. As he does his job, he uses Studio to keep an eye on the KPI's for his process. He has a very limited understanding of what Lumada or Studio is. He has very basic training in using Studio.

  • Mostly concerned with checking for alerts, then following a process for a course of action regarding repair/maintenance
  • Would possibly be alerted by some type of mobile device

Defining the User Permissions

I knew from previous projects, working as a front-end dev, that user permissions are vital to establish early in the project, so that a designer knows what each user can view, and what each user can perform. There is another application in the works under the Lumada umbrella (the GTM name still hasn’t been locked down) that allows admin users to manage Lumada application templates, which contain the logic to query the APIs and visualize the data. After some more meetings about this, I knew the admin persona wouldn’t even need to touch the Studio UI, but rather should only be working in this one other app, to eliminate app-hopping, and building the dashboards in that app for the Studio environment. The Studio information architecture then became a bit more simple: a mostly read-only application for the OT Operator Eric, with some admin management permissions for Manager Mary.

Customizing the Experience

Since the Studio could potentially be used by any industry that wants to collect and visualize data coming from machine sensors, the dashboards needed to be customizable for each client. Our team of engineers chose the open-source project Superset, by Airbnb, to enable a Hitachi consultant (Admin Adam) to configure any arrangement of business intelligence widgets, scoped to an individual user if needed. This level of customization meant that the navigation would be dead simple. I love dead simple.

Design Phase

The navigation pattern I envisioned for this application followed a mobile-first approach, where the user simply drills down by interacting with elements in the dashboard, rather than clicking into an unknown amount of nested dropdown menus. To move back, the user can click/tap the “back arrow”, which takes the user back up one level, or the “home” icon which jumps them back to the top level.
